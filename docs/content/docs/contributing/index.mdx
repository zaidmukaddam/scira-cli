## Run in development

```bash

вќЇ bun run dev -- init  # run a CLI command in dev mode
вќЇ bun run dev -- "what is a Merkle tree?"
```

`bun run dev` runs the CLI directly from `src/` via `tsx` вЂ” no build step needed. Changes to any source file reload automatically.

## Build

```bash
вќЇ bun run build   # compiles to dist/
вќЇ bun link        # registers scira globally so you can call it as `scira`
```

## Run tests

```bash
вќЇ bun test                         # run all tests once
вќЇ bun test --watch                 # watch mode
вќЇ bun test src/tools/workspace     # run a specific file or directory
```

Tests use Vitest (configured in `vitest.config.ts`). No mocks hit the real network вЂ” anything that calls an external service is integration-tested via recorded fixtures or needs credentials in `.scira/.env`.

## Code style

- **TypeScript** throughout вЂ” no `any` unless genuinely unavoidable
- **Prettier** for formatting (run `bun run format` before committing)
- **ESLint** for linting (run `bun run lint`)
- Prefer `async/await` over `.then()` chains
- Keep tool adapters stateless вЂ” they receive config, call a network API, return structured data

## Areas that need help

If you're looking for somewhere to start, these are the most useful contributions:

- **Search provider adapters** вЂ” new backends for `src/tools/search-web.ts` (see the banned list below before you start)
- **Export formats** вЂ” PDF, HTML, or Obsidian-flavored markdown in the `scira export` command
- **TUI improvements** вЂ” keyboard shortcuts, better source display, inline diff for claims
- **Test coverage** вЂ” `src/tools/` is well-covered; `src/agent/` and `src/cli/commands/` need more tests
- **Windows support** вЂ” the TUI currently assumes a POSIX terminal; PRs that improve Windows compatibility are appreciated
- **Documentation** вЂ” typos, missing examples, unclear descriptions

<Callout type="warn" title="Banned search providers">
  Scira does not accept adapters for **Tavily**, **Brave Search**, or **Perplexity**. PRs adding them will be closed. Stick to the supported backends вЂ” Exa, Parallel, and Firecrawl вЂ” or open an issue to discuss a new one first.
</Callout>

## Opening a pull request

1. Fork the repo and create a branch: `git checkout -b feat/my-change`
2. Make your changes, write or update tests, and confirm `bun test` passes
3. Run `bun run lint` and `bun run format`
4. Push and open a PR against `main`
5. Describe what changed and why вЂ” link to any relevant issues

<Callout title="Keep PRs focused">
  One logical change per PR makes review faster and merge conflicts rarer. If you're fixing a bug and also want to refactor something nearby, split them.
</Callout>

## Reporting issues

Open an issue at [github.com/zaidmukaddam/scira-cli/issues](https://github.com/zaidmukaddam/scira-cli/issues). Include:

- Scira version (`scira --version`)
- Operating system and shell
- The exact command or TUI action that triggered the problem
- Any error output (from `~/.scira/logs/` if available)
